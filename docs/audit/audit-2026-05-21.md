**Beleg:**

- `docs/adr/0005-pwa-custom-service-worker.md:42–45`:

2026-05-XX)" mit Begründung des Wechsels, oder Status auf „Superseded by ADR-0007" und ein
neues ADR-0007 mit Titel „Service Worker: kein API-Cache wegen Multi-User-Risiko" anlegen.

#### FIND-006 · C — Plan-Generation ohne Rate-Limit

**Beleg:** `ProgressLab/src/routes/api/plan/generate/+server.ts:5–9`:

```ts
export const POST: RequestHandler = async ({ locals }) => {
  if (!locals.user) throw error(401, 'Nicht angemeldet');
  const plan = await generatePlanTemplates(locals.user.id);
  return json(plan, { status: 201 });
};
```

Keine `checkRateLimit`-Aufrufe.

**Beschreibung:** `generatePlanTemplates` macht mehrere Mongo-Upserts + ein
`Template.deleteMany`. Bei wiederholten Klicks (Doppel-Click oder absichtlich) erzeugt das
DB-Last auf Atlas Free-Tier. Im Single-User-Demo-Setting unkritisch, aber inkonsistent zur
Login-Route (die ist rate-limited).

**Empfehlung:** `checkRateLimit(`plan:${locals.user.id}`, 5, 60_000)` ergänzen.

**Aufwand:** S (< 15 min)

#### FIND-007 · F — Profile-PUT clampt silent statt zu validieren

**Beleg:** Live-Test:

```bash
curl -X PUT -d '{"heightCm":-100,"bodyWeightKg":9999,"trainingDays":999,"experience":"god"}' \
  /api/profile
# Response: {"heightCm":120,"bodyWeightKg":250,"trainingDays":6,"experience":"intermediate",...}
```

`profile-service.ts:113–123` (`clampInteger`, `optionalNumber`) clampt + enumValue fällt
silent auf Default zurück.

**Beschreibung:** Andere Endpoints (`/api/sessions`) weisen ungültige Eingaben mit 400 und
Fehlermeldung ab. Profile clampt still und gibt 200 zurück. User denkt „heightCm = -100
gespeichert", sieht beim Reload aber 120. Inkonsistenz mit dem Rest der App und schlechte
Fehler-Sichtbarkeit (WCAG 3.3.1).

**Empfehlung:** Entweder beide Endpoints angleichen:

- **(a)** Profile-PUT auf strict-mit-400 umstellen wie Sessions.
- **(b)** Sessions-POST auf clamp-mit-Hinweis umstellen wie Profile.

Variante (a) ist üblicher und passt zur expliziten ZHAW-Anforderung „komplexe Validierung
von Usereingaben" — der User merkt, wenn etwas falsch ist.

**Aufwand:** M (30–60 min, weil Tests + UI-Banner mit angepasst werden müssen)
